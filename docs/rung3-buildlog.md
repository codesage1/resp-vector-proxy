Design Q1: Forwarding vs Re-serializing
The proxy must forward the exact original raw bytes. If we chose to re-serialize the parsed resp_value tree, our parser's known limitation (N1: collapsing both Null Arrays and Null Bulk Strings into a single RESP_NULL type) would inject a fatal mutation. A client sending a Null Bulk String ($-1\r\n) would be parsed into RESP_NULL, and if we re-serialized it as a Null Array (*-1\r\n), we would alter the data type of the payload mid-flight. If the upstream server strictly expects a string parameter, this mutation will cause a type error, violating the proxy's core transparency guarantee.

Design Q2: The PROTO_ERR Policy
We must adopt a Fail-Open policy. If the parser returns PROTO_ERR, we must gracefully degrade the connection into a Rung 2 blind passthrough, forwarding the remaining bytes untouched. If we Fail-Closed (dropping the connection), we violate the product promise that "clients see no difference." For example, a human using nc to send an inline PING\n command is issuing perfectly valid Redis protocol, but our strict RESP2/3 parser will flag it as PROTO_ERR. It is not the proxy's job to act as a strict firewall blocking commands it doesn't understand; if a command is truly invalid, it is the upstream Redis server's responsibility to return the error to the client.

Design Q3: The Blob Dimension and Schema State
An 8-byte vector blob is completely ambiguous in isolation. It could be 2 dimensions of FLOAT32 (24 bytes), 8 dimensions of INT8 (81 bytes), or 1 dimension of FLOAT64 (1*8 bytes). The command that dictates this shape is FT.CREATE, specifically the TYPE and DIM arguments inside the index schema definition. Because FT.CREATE is sent long before FT.SEARCH (and potentially on an entirely different TCP connection), this implies that a true inspecting proxy cannot be stateless. To understand search queries, the proxy must maintain global state: it must intercept FT.CREATE commands, extract the schema, and store it in an internal registry so it can cross-reference the index name during future FT.SEARCH commands.
